# TickGit AI 开发文档

## 1. 文档定位

本文档是 TickGit 后续 AI 协作开发的规范文档，也适用于人工开发时的执行标准。

目标：

- 保证后续实现风格一致
- 防止架构漂移
- 防止 AI 在不同轮次中引入互相冲突的实现
- 让“功能开发”和“规范维护”分离

默认规则：

- **先按文档实现**
- **只在必要时更新文档**

---

## 2. 开发前必须阅读

任何新的开发任务开始前，必须先阅读以下文档：

1. `README.md`
2. `docs/ARCHITECTURE.md`
3. `docs/AI_DEVELOPMENT.md`

如果实现与文档冲突，必须明确选择：

- 更新实现以匹配文档，或
- 更新文档以匹配新的正式决策

不允许“先做了再说”，也不允许长期出现文档与实现不一致。

---

## 3. 不可违反的基础规则

## 3.1 包管理器固定为 pnpm

前端与 Tauri CLI 一律使用 pnpm。

允许：

- `pnpm install`
- `pnpm tauri dev`
- `pnpm build`

禁止：

- `npm install`
- `yarn`
- 混用多套前端锁文件

## 3.2 Git 能力只能在 Rust 侧实现

所有 Git 相关逻辑必须放在 Rust 端。

允许：

- Rust 中使用 `std::process::Command`
- 调用系统原生 `git`

禁止：

- 前端直接执行 Git
- 引入 `libgit2` 作为默认方案
- 在前端复制一套 Git 状态判断逻辑

## 3.3 前端只通过统一封装访问后端

调用规则：

- command 统一写在 `src/lib/tauri/api.ts`
- event 统一写在 `src/lib/tauri/events.ts`

禁止：

- 在页面或组件里直接散写 `invoke("xxx")`
- 在多个组件中重复监听相同底层事件且没有封装

## 3.4 修改接口时必须前后端同步

凡是以下任意变更发生时，必须同时检查前后端：

- 字段名称变化
- 字段可选性变化
- 类型变化
- command 参数变化
- event payload 变化

最少要同步检查：

- `src-tauri/src/models.rs`
- `src-tauri/src/commands.rs`
- `src/lib/types.ts`
- `src/lib/tauri/api.ts`
- `src/lib/tauri/events.ts`

## 3.5 不允许无约束扩展范围

如果用户只要求修一个点，不要顺手重做整套架构。

允许：

- 小范围修复
- 局部抽象
- 与当前任务强相关的清理

禁止：

- 借机引入大型状态管理
- 借机整体重写 UI
- 借机替换 Git 执行模型

---

## 4. 当前正式约束

在有新的明确产品决策之前，AI 必须默认遵守以下约束：

- 应用名称为 **TickGit**
- 只支持当前检出分支
- 远端固定为 `origin`
- 推送可用性取决于 origin + upstream + 非 detached HEAD
- 分步提交为单任务模式
- 分步提交暂不支持取消
- Diff 当前为纯文本模式
- 仓库持久化为本地 JSON
- 前端主状态当前由 `src/routes/+page.svelte` 编排

---

## 5. 推荐开发流程

每次任务建议按以下顺序执行：

### 第一步：先读代码，不先猜

至少先检查：

- 相关页面
- 相关组件
- Rust 相关 command / 领域模块
- 相关 DTO

### 第二步：判断任务属于哪一层

按职责归属放置改动：

- UI 展示问题 → `src/lib/components/*`
- 页面编排问题 → `src/routes/+page.svelte`
- 前后端调用问题 → `src/lib/tauri/*`
- Git 逻辑问题 → `src-tauri/src/git.rs`
- 后台任务问题 → `src-tauri/src/jobs.rs`
- 配置持久化问题 → `src-tauri/src/repo_store.rs`

### 第三步：小步修改

优先做最小必要修改，避免一次性改太多文件。

### 第四步：验证

完成后默认至少执行：

```bash
cargo check --manifest-path src-tauri/Cargo.toml
pnpm typecheck
pnpm build
pnpm format:check
```

如果任务非常局部，可说明为何只做部分验证。

### 第五步：判断是否需要更新文档

只有在以下情况下更新文档：

- 架构边界变了
- 规范变了
- 核心约束变了
- 对后续开发方式有长期影响

否则不改文档。

## 6. 代码修改规范

## 6.1 前端规范

- 优先保持组件职责单一
- 展示型组件通过 `props + events` 协作
- 不要在多个组件中复制同一段业务逻辑
- UI 文案统一、明确、桌面应用风格化
- 不要为了抽象而抽象

当前前端状态管理原则：

- 页面级状态留在页面
- 跨页面共享需求尚未出现前，不引入全局状态库

## 6.2 Rust 规范

- command 层保持薄
- Git 命令逻辑集中在 `git.rs`
- 错误统一返回 `AppError`
- 模型尽量进入 `models.rs`
- 后台任务统一由 `jobs.rs` 管理

禁止：

- 在多个文件中重复执行相同 Git 命令逻辑
- 返回结构随意变化
- 直接把底层 stderr 原样拼接成不稳定协议字段

## 6.3 文档规范

新增正式规范时：

- 优先更新现有文档，而不是额外制造很多零散文档
- 文档内容必须是“长期有效”的规则，而不是一次性任务笔记

不应写入规范文档的内容：

- 临时 TODO
- 一次性排障记录
- 与当前仓库无关的泛化建议

---

## 7. 命名与结构规范

### 前端

- 类型名使用 PascalCase
- 变量与函数使用 camelCase
- 组件文件使用 PascalCase
- Tauri DTO 字段在前端使用 camelCase

### Rust

- 模块文件名使用 snake_case
- 结构体与枚举使用 PascalCase
- 函数使用 snake_case
- 对外序列化结构统一 `#[serde(rename_all = "camelCase")]`

### 文档

- 规范文档放在 `docs/`
- 文件名使用清晰语义，优先大写英文常量式命名

---

## 8. 何时必须更新文档

以下情况必须更新至少一份规范文档：

- 应用定位改变
- 技术栈改变
- Git 执行策略改变
- 状态管理方式改变
- 目录分层方式改变
- 后台任务模型改变
- 持久化方式改变
- 公共接口约束改变

以下情况通常不更新文档：

- 修复单个 Bug
- 调整一个组件样式
- 优化局部性能
- 重命名局部变量
- 不影响后续开发方式的小重构

---

## 9. AI 输出要求

AI 在完成开发任务后，应该在总结中明确说明：

- 修改了什么
- 验证了什么
- 哪些限制仍然存在
- 是否更新了文档；如果更新了，为什么要更新

如果因为环境原因无法完成验证，也必须明确说清楚。

---

## 10. 最终原则

TickGit 后续开发遵循四条最高优先级原则：

1. **先符合文档，再扩展功能**
2. **先保证边界清晰，再追求抽象优雅**
3. **先保证可验证，再追求实现速度**
4. **先保持一致性，再做局部优化**

本文档是 TickGit 的 AI 开发规范基线。
