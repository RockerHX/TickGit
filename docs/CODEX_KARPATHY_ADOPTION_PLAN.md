# Codex × Andrej Karpathy Skills 采用计划

## 1. 文档目的

https://github.com/forrestchang/andrej-karpathy-skills

本文档用于规划 TickGit 项目中对 `andrej-karpathy-skills` 思路的引入方式，目标是：

- 先在真实项目中低成本试用
- 再逐步抽象为可复用的 Codex skill
- 最后视验证结果决定是否做成通用 plugin，并向上游仓库提交 PR

本文档是**分阶段采用计划**，不是当前项目的正式长期规范。正式开发规范仍以以下文档为准：

1. `README.md`
2. `docs/ARCHITECTURE.md`
3. `docs/AI_DEVELOPMENT.md`

---

## 2. 背景判断

`andrej-karpathy-skills` 的核心价值，不是提供某个框架能力，而是提供一套更稳健的 AI 编码行为约束，重点包括：

- 先理解问题，再开始编码
- 优先简单方案，而不是过度设计
- 只做与当前目标直接相关的小范围修改
- 在编码前明确目标、验证方式与完成标准

这类内容本质上更像：

- 行为规范
- 执行方法
- 任务工作流约束

而不是产品功能代码。

对 Codex 来说，这套内容具备天然适配性，因为 Codex 本身就支持：

- `AGENTS.md`：长期默认生效的规范
- `SKILL.md`：按需触发或隐式触发的任务能力
- `.codex-plugin/plugin.json`：分发与安装格式

当前判断是：

- 这套思想**适合 Codex**
- 但需要结合 Codex 的原生机制来落地
- 不应一开始就做成“大而全”的通用方案

---

## 3. 总体策略

采用策略分为三个阶段：

1. **先在 TickGit 本地试点**
2. **再抽成 TickGit 内部可复用的 Codex skill**
3. **最后视效果做成通用 plugin / upstream PR**

原则：

- 先验证，再抽象
- 先局部使用，再公开分发
- 先低成本集成，再考虑通用包装

---

## 4. 阶段计划

## 4.1 Phase 1：仓库内本地试点

### 目标

在 TickGit 中以最小成本验证这套规范是否真正提升 Codex 协作开发质量。

### 形式

优先通过仓库级 `AGENTS.md` 引入 Karpathy 风格行为规则，使其作为默认工作方式生效。

### 交付物

- 本文档：`docs/CODEX_KARPATHY_ADOPTION_PLAN.md`
- 仓库级 `AGENTS.md`（新增或补充 Karpathy 风格约束）
- 一轮真实任务试用记录

### 试点范围建议

优先挑选以下任务类型进行试点：

- 小范围 UI 调整
- 现有逻辑上的局部 bug 修复
- 前后端接口字段同步类改动
- 小型交互改进

暂不建议一开始就拿以下任务做验证：

- 大范围重构
- 架构迁移
- 跨多个模块的大型能力新增

### 执行要求

在试点任务中，要求 Codex 默认遵守以下工作方式：

- 先读文档和相关代码，不先猜
- 先说明理解、范围、假设与验证方式
- 优先做最小必要修改
- 不借题发挥扩大改动范围
- 完成后明确说明验证结果与残余风险

### 成功标准

满足以下条件即可认为 Phase 1 有价值：

- 至少 2 到 3 个真实任务中表现稳定
- 明显减少“先猜后改”或“顺手乱改”的情况
- 修改范围更集中
- 输出更容易 review
- 没有明显增加日常协作负担

### 暂停条件

如果出现以下情况，应暂停继续抽象：

- 规范明显过重，导致任务推进变慢
- 指令过于抽象，实际执行不稳定
- 与现有 TickGit 规范冲突较多
- 很难在真实任务中带来可感知收益

---

## 4.2 Phase 2：抽成仓库内可复用 skill

### 目标

将 Phase 1 中验证有效的内容，提炼为可在 Codex 中显式调用的 skill。

### 形式

在仓库内新增：

```text
.agents/skills/karpathy-guidelines/SKILL.md
```

必要时可补充：

- `references/`
- 示例调用说明

### 设计原则

此阶段要明确区分 `AGENTS.md` 与 skill 的职责：

- `AGENTS.md`：默认长期生效的底线规范
- `SKILL.md`：在复杂任务中显式增强的工作流

也就是说，skill 不应只是重复 `AGENTS.md`，而应更强调：

- 如何拆解复杂任务
- 如何先定义验收标准
- 如何收敛范围
- 如何控制改动粒度

### 交付物

- `.agents/skills/karpathy-guidelines/SKILL.md`
- 简短使用说明
- 至少一次显式 skill 调用验证

### 成功标准

- skill 能在 Codex 中稳定调用
- 内容已不强依赖 TickGit 私有上下文
- 对复杂任务相比单纯依赖 `AGENTS.md` 有额外收益

---

## 4.3 Phase 3：通用化与上游贡献

### 目标

将已验证的 Codex 适配方案整理为可分发成果，并评估是否向上游仓库提交 PR。

### 形式

根据实际验证结果，选择以下一种或多种方式：

1. 在本地或独立仓库中制作 `.codex-plugin/plugin.json`
2. 整理 Codex 使用说明
3. 向上游仓库提交最小可接受 PR

### 推荐最小 PR 范围

优先做小而清晰的贡献，不做过度包装。建议 PR 内容包括：

- 增加 Codex 使用说明
- 增加 `.codex-plugin/plugin.json`
- 说明如何作为 `AGENTS.md` 使用
- 说明如何作为 Codex skill 使用

### 不建议一开始做的事

- 一次性引入大量平台兼容层
- 过早设计复杂 marketplace 分发方案
- 在没有真实使用反馈前大规模抽象

### 成功标准

- Codex 用户能看懂如何安装和使用
- 适配方案与原仓库理念一致
- PR 变更范围小、可审查、可维护

---

## 5. 为什么不单独立即开新项目

当前阶段不建议立即单独开新项目，原因如下：

- 还没有经过真实任务验证
- 还不知道哪些内容真正适合 Codex
- 现在做独立项目，容易先包装、后验证
- 在 TickGit 里试点能获得最真实的上下文与反馈

因此当前推荐路径是：

**先在 TickGit 验证 → 再抽象 → 再分发**

---

## 6. 当前执行顺序

当前建议按以下顺序推进：

1. 完成本文档
2. 在仓库中增加或补充 `AGENTS.md`
3. 选择一个小任务作为第一轮试点
4. 记录试点观察结果
5. 再决定是否进入 skill 抽象

当前不进入：

- plugin 开发
- marketplace 分发
- 上游 PR

这些工作只有在 Phase 1 和 Phase 2 验证有效后再做。

---

## 7. 后续对话使用方式

后续如开启新对话，应将本文档作为当前阶段的执行依据，并明确：

- 当前处于哪个 phase
- 本轮任务目标是什么
- 本轮是否只是试点，不做抽象
- 是否需要更新 `AGENTS.md` 或新增 skill

建议在新对话中优先说明：

> 按 `docs/CODEX_KARPATHY_ADOPTION_PLAN.md` 当前阶段推进，只做本阶段目标，不提前做后续阶段工作。

这样可以避免一次性扩张范围。

---

## 8. 当前结论

对于 TickGit 来说，最合理的路线不是立即把 `andrej-karpathy-skills` 做成完整 Codex 插件，而是：

1. 先作为仓库级行为规范试点
2. 再抽成可复用 skill
3. 最后再考虑 plugin 与 upstream PR

这条路线更稳、更轻，也更符合“先验证、后抽象”的工程方式。
