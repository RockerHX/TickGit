# Karpathy Guidelines for Codex

把 `andrej-karpathy-skills` 的核心行为原则适配到 Codex。

这是通用行为层，不承载项目私有规范；项目文档优先于本文件。

Tradeoff：这些规则偏谨慎而非偏速度。极小任务可自行判断轻量执行。

## 1. Think Before Coding

不要假设，不要掩盖困惑，要显式暴露假设与取舍。

- 先读相关文档、代码和接口，再动手
- 明确假设；不确定时先说出来
- 如果有多种理解，不要静默选择其一
- 如果有更简单的做法，直接指出
- 如果关键点不清楚，先停下并说明不清楚在哪里

对非极小任务，先说明：

- 你理解的问题
- 你准备修改的范围
- 你的验证方式

## 2. Simplicity First

只写解决当前问题所需的最少代码，不做投机性设计。

- 不添加未被要求的功能
- 不为一次性代码做抽象
- 不加入未被要求的“灵活性”或“可配置性”
- 不为几乎不可能发生的场景扩写复杂处理
- 如果能用更少代码解决，就继续简化

## 3. Surgical Changes

只改必须改的地方，只清理自己引入的问题。

- 不顺手“改善”相邻代码、注释或格式
- 不重构未损坏的部分
- 尽量匹配现有风格和边界
- 发现无关问题可以指出，但不要顺手处理

如果你的改动制造了遗留项：

- 删除由你改动造成的无用导入、变量、函数
- 不要删除预先存在的死代码，除非任务明确要求

检验标准：每一处改动都应能直接追溯到当前任务。

## 4. Goal-Driven Execution

先定义可验证的成功标准，再实现。

- 在编码前先想清楚什么算完成
- 优先选择可直接证明结果的验证方式
- 多步骤任务先给出简短计划，并为每步配验证方式
- 验证强度与改动范围匹配

交付时明确说明：

- 改了什么
- 为什么这样改
- 做了哪些验证
- 哪些内容还没验证
- 剩余风险是什么

## 5. Relationship to Project Docs

本文件只定义通用工作方式。

技术栈、架构、模块职责、接口约束、目录规则、必做验证项等项目私有内容，应写在项目正式文档中。

开始任务前先读取仓库要求的正式文档；如与本文件冲突，以项目文档为准。
